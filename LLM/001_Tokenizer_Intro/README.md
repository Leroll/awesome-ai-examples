# 让模型理解文字：Tokenizer 原理探究


## 1. 从“汉字”到“Token ID”再到“Embedding”的全过程

整个过程可以分为两大阶段：**训练阶段（形成词表）** 和 **推理阶段（处理输入文本）**。

下面我以汉字 **“我”** 为例，为你一步步拆解从“汉字”到“BPE分词”，再到“Embedding（向量）”的全过程：

---

### 1.1：训练阶段 —— `tokenizer.json` 是怎么来的？

在这个阶段，模型要从海量的语料中学习出一个“词表（Vocabulary）”和“合并规则（Merges）”。

#### 第一步：汉字转 UTF-8 字节（Bytes）
计算机底层不认识“我”，它只认识 0 和 1。在主流的文本编码（UTF-8）中，**一个常用的汉字通常占用 3 个字节**。
*   汉字：`我`
*   UTF-8 字节（十六进制）： `E6 88 91`
*   对应十进制：`230`, `136`, `145`

#### 第二步：字节映射（Bytes to Latin 字符）
这是 BPE（Byte-Level BPE）非常巧妙的一步。
因为传统的 BPE 算法是处理字符串的，如果遇到生僻字，词表里没有，就会输出 `<unk>`（未知字符）。为了彻底消灭 `<unk>`，研究人员决定**直接对底层的 256 个字节进行 BPE 合并**。
但在很多编程语言和算法库里，强行处理那些不可见的控制字节（比如 136, 145）会报错。所以，Tokenzier 会使用一个固定的隐射表，**把 256 个字节一一映射到 256 个可见的扩展拉丁字符上**（比如你在这个 JSON 里看到的 `Ġ`, `Ċ`, `å`, `æ` 等）。
*   假设映射规则是：`230` -> `æ`, `136` -> `Ī`, `145` -> `ĳ`
*   此时，“我”在 Tokenizer 眼里就变成了字符串 **`æĪĳ`**。

#### 第三步：BPE 统计与合并（Merges & Vocab）
现在，Tokenizer 拿着海量的被替换成拉丁字符的语料开始统计。
1.  **初始状态**：词表里只有 256 个基础字节字符（所有的汉字目前都被切得稀碎，变成离散的字节）。
2.  **寻找最高频对**：它发现 `æ` 和 `Ī` 总是挨在一起出现，于是把它们合并成 `æĪ`，加入词表。
3.  **继续合并**：接着它发现 `æĪ` 和 `ĳ` 也总是挨在一起（因为“我”是一个极高频的字），于是合并成 `æĪĳ`，并加入词表，分配一个固定的数字 ID（假设是 `16000`）。
4.  如果是更高频的词（比如“我们”），甚至会把代表“我”的三个字节和代表“们”的三个字节一起合并，变成一个由6个字节组成的 Token！
*   **结果**：在这个过程中，常见的中文字/词被“重组”并加入到 `vocab` 中。这就是为什么你的 `tokenizer.json` 里全是看似乱码的字符组合，因为它们记录的是 **高频的字节组合**。

---

### 1.2：推理阶段 —— 你打出“我”字后发生了什么？

当你向 Qwen 发送一句 Prompt，比如：“**我**”，模型会跑以下流程：

#### 第四步：输入文本转字节与映射（Encoding）
你输入的“我”，被 Tokenizer 翻译成底层的字节，并套上“马甲”：
*   “我” -> `E6 88 91` -> **`æĪĳ`**。

#### 第五步：切词与查表（Tokenization）
Tokenizer 拿着字符串 `æĪĳ`，去查它之前存好的 `tokenizer.json`（规则和词表）：
*   它发现词典里正好有 `æĪĳ` 这个完整的条目。
*   它提取出对应的整数 ID，比如 **`16000`**。
*(注：如果此时你输入了一个非常极其罕见的生僻字，假设它对应的字节组合并没有在训练时被合并入词典，Tokenizer 就会把它拆成 2 个或 3 个独立的基础字节 ID 传给模型，以此保证模型永远不会遇到“无法识别”的字。)*

#### 第六步：ID 转 Embedding (查向量表)
走到这一步，文字已经变成了纯粹的数字 ID `16000`。
大模型内部的第一层，是一个巨大的 **Embedding矩阵 (词向量表)**。
*   假设 Qwen 的词表总大小是 150000 左右，模型的隐藏维度（Hidden Size）是 4096。
*   这个矩阵的尺寸就是 `[150000, 4096]`。矩阵里的每一个数字都是在预训练阶段被反复调整、学出来的“权重”。
*   模型直接去这个矩阵里提取第 **`16000`** 行。
*   提取出来的是一个长度为 4096 的数组（比如 `[0.012, -0.34, 1.25, -0.08, ...]`）。

#### 第七步：输入 Transformer 层计算
此时，“我”这个字已经不复存在，取而代之的是含有“我”的语义、语法信息的致密浮点数向量（4096 维）。这个向量会被加上位置编码（告诉模型这是句子里的第几个字），然后送入层层叠叠的 Self-Attention 机制中进行推理计算。

---

### 简单总结一下一条数据的奇妙旅程：

屏幕上的 **“我”**
 ⬇️ (UTF-8 编码)
变成三个字节：**230, 136, 145** 
 ⬇️ (字节级 BPE 映射) 
穿上英文字符马甲变成：**`æĪĳ`**
 ⬇️ (BPE 分词器查表) 
匹配到对应的 Token ID：数字 **16000**
 ⬇️ (查模型的 Embedding 权重表)
变成一个致密的浮点向量：**[0.12, -0.55, 1.23, ...]**
 ⬇️ (进入大模型的脑子)
开始理解语义并预测下一个向量背后的含义并预测下一个 Token！

## 2. token 与 汉字、英文单词比例关系

### 2.1 直接统计 vocab.json 是否可行？ 

这是语言学中经典的 **Type（词表类型） vs Token（实际语料）** 的问题。

1. **词典不等于实际交流**：
   假设你的 `vocab` 里有 10 万个 token。其中有很长的 token（比如国产模型里的 `中华人民共和国` 算1个 token，包含7个汉字），也有很短的 token（比如 `的` 算1个 token，1个汉字）。
   如果你直接拿 `vocab` 求平均，你给了 `中华人民共和国` 和 `的` **完全相同的权重（1:1）**。
2. **长尾效应（齐夫定律）**：
   但在真实的文本中，`的`、`我`、`是` 这些单字 token 出现的频率可能高达百万次，而 `中华人民共和国` 可能只出现几次。
3. **未完全合并的汉字（Byte-fallback）**：
   很多生僻字在 `vocab` 里根本不存在，它们被拆解成了 2~3 个基础的字节（Byte） token。单纯看 `vocab` 是统计不到这些组合的。

**结论**：直接算 `vocab` 的平均长度，就像“把马云的资产和你的资产加起来平均”，毫无实际参考价值。

---

### 唯一正确的计算方法：语料级评测

要真正知道“1个Token代表多少字”，必须拿一段**足够大且真实的文本（Corpus）**，用分词器切分一下，然后做除法：
* **公式（中文）** = `文本中的总汉字数` ÷ `切分后产生的总 Token 数`
* **公式（英文）** = `文本中的总英文单词数` ÷ `切分后产生的总 Token 数`

### 2.2 目前行业内的“真实统计数据”究竟是多少？

为了帮你省去跑几百兆语料的时间，我为你总结了目前主流模型在**大规模真实语料**测试下，公认的、未被污染的统计数据结果：

#### 1. 英文（各大模型差异不大）
因为大模型几乎都是以英文为主，所以对英文的压缩率非常稳定。
* **1 个 Token ≈ 0.75 个英文单词**
* 也就是 OpenAI 官方给出的经验法则：**100 tokens ≈ 75 words**（或者说一个英文单词约等于 1.3 个 Token）。

#### 2. 中文（不同模型天差地别）
这是被污染最严重的地方，因为不同模型的中文词表大小完全不一样。

* **GPT-3 / GPT-3.5 早期 (p50k_base)**
  * **1 个 Token ≈ 0.3 ~ 0.5 个汉字**（极度拉胯）。
  * 也就是说 1个汉字通常会被切成 2个以上的 Token。因为当时它的中文词表极小，只能回退到字节层面。
* **GPT-4 / Claude 3 (cl100k_base 等现代国际模型)**
  * **1 个 Token ≈ 0.6 ~ 0.8 个汉字**。
  * 大多数常见单字是 1个 Token，两字词汇偶尔被合为 1个 Token，但生僻字仍然会占用 2-3个 Token。
  * **经验法则**：对于 GPT-4，你写了 1000 个汉字，大概会消耗掉 **1200 ~ 1500 个 Token**。
* **国产优秀模型 (如 Qwen、DeepSeek、ChatGLM)**
  * **1 个 Token ≈ 1.2 ~ 1.8 个汉字**（遥遥领先）。
  * 国产模型针对中文进行了疯狂的扩表（比如 Qwen 词表高达 15万），大量的双字词（`我们`）、四字成语（`不以为然`）、甚至专有名词（`人工智能`）都直接是一个 Token。
  * **经验法则**：对于国产模型，你写了 1000 个汉字，大概只消耗 **600 ~ 800 个 Token**。

**总结给你的建议：**
不要去看 `tokenizer.json`，直接写一个简单的脚本。拿几十篇你的目标场景的文章（比如如果是医学，就拿医学论文），喂给你选定的 `tokenizer.encode()`。统计出来的 `总字数 / 总Token数` 比例，就是最完美、针对你的业务最精准的答案。
